Model Settings

Tune the assumptions behind the challenge recommendation model.

Core Formula Expected Value = (success chance x overturn value) - (miss chance x challenge cost)

These controls change how aggressive the model is before it turns the math into player-facing levels.

Success

Higher rates make challenges easier to justify.

Cost

Higher costs protect challenge inventory.

Thresholds

Higher bars keep levels more conservative.

Expected Success Probability

Estimated chance the challenge is overturned. Default inputs are the public success rates for all catchers and all batters. If the user thinks a specific catcher or batter has more or less projected success, adjust the input.

Challenge Cost

How expensive it is to lose a challenge. The 5.5 WP point default is anchored to neutral, middle-game future opportunity samples; inventory factors make the last challenge more costly than the first.

Remaining Game

These weights scale the opportunity cost of spending a challenge. Early innings are higher because more future high-value challenge chances remain; late innings are lower because unused challenges have less time to matter.

Level Thresholds

Rules for upgrading from Closed to Full Count, Deep Count, or Open. Higher values demand stronger evidence.

Current Win Probability Cost Curve

Scales challenge cost by the team's current win probability. Lower multipliers loosen the model because unused challenges have less future value.

What We Did

The decision engine uses expected value: challenge value equals success probability times the win-probability swing of a successful overturn, minus failure probability times the opportunity cost of losing the challenge.

Expected Value = p * V - (1 - p) * C

p is expected challenge success probability. The v1 defaults are 47% for batters and 59% for catchers, with a manual override for team or player reads.

V is the win probability swing between the umpire's call and the corrected call. The app asks Savant's count-aware Game Strategy Explorer endpoint for state values, so inning, half, score, outs, runners, and count all matter.

C is the marginal cost of burning a challenge. The v1 proxy is 5.5 win probability points times inventory, remaining-game, and current-WP multipliers.

Closed

Fallback when Full Count does not clear. Only unmistakable misses.

Full Count

Clears when median expected value of 3-2 opportunities is at least zero.

Deep Count

Clears when full-count plus two-strike or three-ball situations have nonnegative median expected value and at least a 55% positive rate.

Open

Clears when all counts have nonnegative median expected value and at least a 65% positive rate.

The public endpoint is useful for a prototype, but a team-grade version should cache responses and let clubs plug in their own win expectancy model.

How To Use

Before a plate appearance, set the game state, the challenge side, and the number of challenges remaining. The recommendation is the player-facing gate for that plate appearance.

  1. My Team is the side receiving the recommendation. For catcher recommendations, My Team is the fielding team.
  2. Set expected challenge success rates in Model Settings when the staff has a better team or player read.
  3. Give hitters the simple gate only. Catchers can receive the count detail table when the staff wants more precision.
  4. Use Guide Export after settings changes so the staff handout matches the model.

Open does not mean challenge randomly. It means the count restriction is open if the player has a strong read that the call missed.
